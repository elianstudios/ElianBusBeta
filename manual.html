Chapter 1Welcome to ElianBus

ElianBus is a tiny message hub that lets every app on this Mac talk to every other app through one central point — with a complete, human-readable record of all traffic on disk.

Think of it as a post office with a photocopier. Every app drops its letters at the same counter. The clerk photocopies each letter into a bound ledger first, then hands copies to everyone who asked for that kind of mail — and, if you have marked that kind of letter as important, also texts a copy to your phone.

Apps never talk to each other directly. NetworkDashboard does not know ElianCron exists; your phone does not know NetworkDashboard exists. Each of them knows exactly one address: 127.0.0.1:9900. Connect a new app once, and it can immediately hear — and be heard by — everything else, including apps that will not be written for another year.

1.1 — The Three Guarantees

  • Log-first. Every message is appended to bus.jsonl before anyone is notified. If it's not in the log, it didn't happen. The log is the source of truth, not a side effect.
  • One shape, forever. Every message is the same four-field envelope (Chapter 4). No app ever needs to learn another app's format.
  • Humans are first-class. A message you type by hand flows through the same log, the same fan-out, and the same phone filter as any machine message. Nothing is special-cased.

1.2 — The Shape of the System

 NetworkDashboard ──publish──▶ ┌───────────────────┐
 ElianCron (round 2) ◀──WS──▶  │      ElianBus      │──▶ bus.jsonl   (everything, forever)
 any future app ──────────▶    │  127.0.0.1:9900    │──▶ WS subscribers (live fan-out)
 you (curl / web page) ───▶    └───────────────────┘──▶ ntfy ──▶ 📲 your phone (ticked topics)

The hub itself is deliberately dumb: about 180 lines of boring Node.js. It logs, it fans out, it forwards. All intelligence stays in the apps. There is no database, no broker, no framework — one file, one dependency (ws), one port.

Chapter 2Quick Start

The hub is already installed as a LaunchAgent and starts at login. Five minutes, five steps:

2.1 — See it breathe

curl -s localhost:9900/topics

A JSON list of every topic ever seen. If this answers, the bus is alive.

2.2 — Open the control room

Open http://localhost:9900. Live feed at the center, phone-push picker above it, inject form at the bottom.

2.3 — Publish your first message

curl -d '{"from":"human","topic":"test/hello","data":{"msg":"my first message"}}' \
     localhost:9900/pub

Watch it appear in the live feed the same instant, and in the ledger:

tail -1 bus.jsonl

2.4 — Wire up your phone

  1. Install the ntfy app (Play Store / F-Droid) on your phone.
  2. On the web page, press Generate — this creates a long secret topic name.
  3. In the ntfy app: Subscribe to topic → paste that name.
  4. Flip the switch on, press Send test. Your phone buzzes.

2.5 — Choose what reaches you

In the Topics card, tick 📲 phone next to each topic you want forwarded. network/intruder and system/warning are ticked by default. Everything else stays on the Mac unless you say otherwise.

Chapter 3Connecting an App — The Minimum

The entire integration contract is: send one HTTP POST. If your app can run curl, it is already compatible.

3.1 — Level 1: Speak (one line, any language)

This is the minimum. One fire-and-forget POST wherever something notable happens:

# shell — the whole adapter
curl -s --max-time 2 -d '{"from":"my-app","topic":"my-app/event","data":{"msg":"it happened"}}' \
     localhost:9900/pub >/dev/null 2>&1 &
# python
import urllib.request, json
def bus(topic, msg):
    try:
        req = urllib.request.Request("http://127.0.0.1:9900/pub",
            json.dumps({"from":"my-app","topic":topic,"data":{"msg":msg}}).encode())
        urllib.request.urlopen(req, timeout=2)
    except Exception: pass   # hub down = shrug, never crash the app
// node
fetch("http://127.0.0.1:9900/pub", { method:"POST",
  body: JSON.stringify({ from:"my-app", topic:"my-app/event", data:{ msg:"it happened" } })
}).catch(()=>{});
House rule: publishing must never block or crash the caller. Short timeout, swallow errors, background it. The bus is an observer of your app, not a dependency. (NetworkDashboard's bus-pub.sh is the reference implementation.)

3.2 — Level 2: Listen (one WebSocket)

To react to other apps, hold a WebSocket open and say which topics you care about:

// node — a complete subscriber
const WebSocket = require("ws");
const ws = new WebSocket("ws://127.0.0.1:9900/ws");
ws.on("open", () => ws.send(JSON.stringify({ type:"sub", topics:["presence/*"] })));
ws.on("message", line => {
  const e = JSON.parse(line);          // {ts, from, topic, data}
  console.log(e.topic, e.data.msg);
});

Reconnect on close (a 2-second retry loop is plenty), and on reconnect call /replay?since=<last ts you saw> to catch up on anything you missed — the log remembers so your app doesn't have to.

3.3 — Level 3: Converse

Full participation is just both halves at once: subscribe to command topics (my-app/run), publish result topics (my-app/done). This is exactly what the round-2 ElianCron adapter does with cron/run → job/completed.

LevelEffortYour app can…
1 · Speak1 linebe seen: log, live feed, phone push
2 · Listen~10 linesreact to any other app's events
3 · Converse~25 linesbe driven by the bus and report back

Chapter 4The Envelope

Every message on the bus — machine or human — is this one JSON shape:

{
  "ts":    "2026-07-16T01:00:00.000Z",   // stamped by the hub if you omit it
  "from":  "network-dashboard",          // REQUIRED — who is speaking ("human" for you)
  "topic": "presence/arrival",           // REQUIRED — what this is about, a/b/c path
  "data":  { "msg": "Phone is home" }    // anything; defaults to {}
}
FieldRules
tsISO-8601. Omit it — the hub stamps arrival time. Replay's since= compares against this.
fromNon-empty string. Your app's stable name, kebab-case. Hand-injected messages use human.
topicNon-empty string, slash-separated path. Missing/empty → the hub rejects with 400.
dataAny JSON. Two soft conventions: data.msg = human-readable one-liner (used as the phone-push body and the feed display), data.title = optional push title.
Write the msg for a human. Whatever ends up in data.msg is what future-you reads in the feed, the log, and on your phone's lock screen. "Phone is home" beats {"idk":"mac:f8:…","n":0}.

Chapter 5Topics

Topics are labeled channels. They are not declared anywhere — the first publish on a topic brings it into existence, and it automatically appears on the web page with its own phone checkbox.

5.1 — Current Conventions

TopicPublisherMeaning
presence/arrivalnetwork-dashboarda trusted device came online
presence/departurenetwork-dashboarda trusted device left
network/intrudernetwork-dashboardunknown device on the Wi-Fi
automation/firednetwork-dashboardan automation rule ran (ok / ERR)
system/warninganyonesomething broke and needs a human
bus/testhubtest pushes from the Send-test button
job/completed · job/failedeliancronreserved for round 2
cron/runanyonereserved for round 2 — ask ElianCron to run a job

5.2 — Wildcards

Subscriptions, replay filters, and phone-forward rules all use the same matcher:

PatternMatches
presence/arrivalexactly that topic
presence/*presence/arrival, presence/departure, and anything deeper (presence/a/b)
#  or  *everything

The wildcard is trailing only — */arrival is not a pattern, it's a literal topic name that will never match anything.

5.3 — Naming Guidance

  • First segment = domain (presence, network, job), second = event, lowercase.
  • Name events in the past tense for facts (job/completed), imperative for commands (cron/run).
  • network/* and system/* are pushed to the phone at high priority — reserve them for things worth a loud buzz.

Chapter 6The Hub Interfaces

6.1 — HTTP Endpoints

EndpointMethodFunction
/pubPOSTPublish one envelope. Returns {"ok":true} or 400 with a reason. Body limit 1 MB.
/replayGETRead the log back. Params: topic (wildcard, default #), since (ISO ts, inclusive), limit (last N).
/topicsGETEvery distinct topic ever seen (seeded from the log at boot).
/configGET / POSTRead or partially update bus-config.json. POST merges: send only {"forward":{"x/y":true}} to change one rule.
/test-pushPOSTSend a test message to the phone, ignoring forward rules (still requires push enabled + a topic set).
/GETThe web page.
/manualGETThis document.

6.2 — WebSocket Protocol (ws://127.0.0.1:9900/ws)

Frames you send:

{"type":"sub", "topics":["presence/*", "job/*"]}   // replaces your subscription list
{"type":"pub", "from":"my-app", "topic":"a/b", "data":{...}}

Frames you receive: raw envelope lines, exactly as written to the log. Until you send a sub frame you receive everything (#) — convenient for dashboards, noisy for adapters, so subscribe early.

Chapter 7The Log — bus.jsonl

bus.jsonl is an append-only text file, one JSON envelope per line, in arrival order. It is written before fan-out and before the phone forward — the ledger never lies by omission.

# watch the whole ecosystem live
tail -f bus.jsonl

# everything that happened today about presence
curl -s 'localhost:9900/replay?topic=presence/*&since=2026-07-17T00:00:00'

# count messages per topic (jq)
jq -r .topic bus.jsonl | sort | uniq -c | sort -rn

Because the log is the source of truth, a subscriber that was offline hasn't lost anything — it reconnects, replays since its last seen ts, and is whole again. Apps can be restarted fearlessly.

The log grows forever. There is no rotation yet (see Chapter 12). At this ecosystem's message rates that's years of headroom, but if you ever bulk-publish, remember every byte is kept.

Chapter 8Phone Push (ntfy)

The hub's only outward-facing feature. After logging and fan-out, each message is checked against your forward rules; matches are POSTed to ntfy.sh, which delivers them to the ntfy app on your phone — anywhere in the world, mobile data included. The phone does not need to be on your network. If it's briefly offline, ntfy holds messages ~12 hours.

8.1 — Configuration (bus-config.json)

{
  "ntfy":    { "enabled": true, "server": "https://ntfy.sh", "topic": "elian-x7k2…" },
  "forward": { "network/intruder": true, "system/warning": true, "presence/*": false }
}

Forward keys accept the same trailing wildcards as subscriptions. Edit from the web page, via POST /config, or by hand — the file is re-read on every message, so changes are instant, no restart.

8.2 — Priorities

Topic familyPriorityPhone behaviour
network/* · system/*4 (high)loud, bypasses some quiet settings, ⚠️ tag
everything else3 (default)normal notification, 📨 tag

8.3 — Privacy, Honestly

  • Messages transit the public ntfy.sh server, cached ~12 h, then deleted. No account exists; the topic name is the only secret. That is why Generate makes a long random one. Don't share it, don't reuse it elsewhere.
  • Content is low-sensitivity by design ("Phone is home", a MAC address) — but it is your home's rhythm. If that ever feels like too much, self-host: run ntfy serve in Docker, change the server field, done. Same app, zero cloud.
  • Free-tier rate limits (hundreds of messages/day) are far above this ecosystem's traffic.

Publishes through its bus-pub.sh (2-second cap, backgrounded, silent when the hub is down — detection loops can never stall on the bus). Hooked at every gated announce site, so everything is already de-duplicated and cooldown-gated upstream:

A WebSocket adapter in the Electron main process: subscribes to cron/run → calls engine.runNow(jobId); publishes job/completed / job/failed from the job-finish hook. The source has been mapped; the work is scoped and waiting for the word.

Chapter 12What ElianBus Can NOT Do (Yet)

Read this chapter before trusting the bus with something important. Every limit below is a deliberate round-1 trade-off, not an accident — and each has a stated workaround or a planned fix.

LimitWhat it means in practice
No delivery guaranteesFan-out is fire-and-forget. A subscriber that is offline misses live messages; nothing is queued per-client, there are no ACKs, no redelivery. Workaround: on reconnect, call /replay?since=… — the log is the queue.
No request/responsePure pub/sub. You cannot "call" an app and await its answer; you publish a command topic and separately subscribe to its result topic. Correlate by putting an id in data.
No auth, no TLSAnything running on this Mac (any user process) can publish, subscribe, and read/change the push config. Localhost-only binding is the entire security model. Fine for a personal machine; do not port-forward 9900.
Localhost onlyOther devices on the LAN cannot reach the bus. The ONLY off-Mac output is the one-way ntfy push. (A LAN/remote story would need auth first.)
No log rotationbus.jsonl grows forever, and /replay reads the whole file into memory. Years of headroom at current rates, but a bulk publisher would bloat both. SQLite/rotation is the planned fix if it ever hurts.
No schema validationThe hub checks from + topic exist and the body parses — nothing else. A publisher sending garbage data ships garbage to every subscriber. Conventions (Chapter 4) are enforced by discipline, not code.
No dedup / rate limitingThe hub forwards every message it accepts. A flapping publisher = a buzzing phone. Today all publishers gate themselves (cooldowns, alerted-macs); a new app must do the same.
Push has no retry queueThe ntfy forward is one attempt with an 8-second cap. Mac offline → that push is gone (the message itself is still logged and fanned out). No batching, no digest mode either — every match is one notification.
Trailing wildcards onlypresence/* works; */arrival or a/*/c do not exist.
Single hub, single pointIf the hub is down, nothing flows (publishers shrug and drop). The LaunchAgent respawns it in seconds, and publishers are built to tolerate the gap — but there is no clustering, no failover.
Config changes are not eventsEditing forward rules / ntfy settings is not itself published to the bus, so there's no audit trail of who changed what beyond file mtime.
ElianCron not wired yetcron/run, job/completed, job/failed are reserved names — publishing them today reaches the log and your phone, but no engine acts on them. That is round 2.
